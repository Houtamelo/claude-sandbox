# claude-sandbox — Design

Date: 2026-05-10
Status: Approved, pending spec review

## 1. Goal

Run Claude Code in a sandboxed Linux environment where Claude has full `sudo` and unrestricted ability to install packages, run scripts, and modify its working dir, while being structurally unable to damage the host's filesystem, packages, or system configuration. Personal user instructions, skills, and agent definitions defined on the host must be transparently visible to Claude inside.

## 2. Threat model

Severity assigned by user (houtamelo) during brainstorming:

| Risk | Severity |
| --- | --- |
| A. Filesystem damage outside the project (`~/.config`, dotfiles, system files) | High |
| B. Package/system mutation (`zypper install`, systemd, kernel modules) | High |
| C. Credential/network exposure (`~/.ssh`, `~/.aws`, browser cookies, env tokens) | Medium |
| D. Resource/process runaway (fork bombs, disk fill) | Low |

A + B as primary drivers select for **a rootless container** (Podman). C is addressed by mount policy (default deny + per-project opt-in). D is not addressed at the runtime layer; user accepts.

## 3. Non-goals

- macOS / Windows support. Linux only (target: openSUSE Tumbleweed host; container OS is Debian).
- Multi-user containers, remote container hosts.
- GUI/TUI dashboard (out-of-scope; that is the schmux problem space).
- VS Code `devcontainer.json` compatibility.
- Auto-update of base image.
- Defense against in-container prompt-injection beyond config read-only + bounded mounts.
- Container resource policy beyond simple `limits.memory` / `limits.cpus` passthrough.

## 4. Architecture

### 4.1 Runtime

- **Rootless Podman.** Container UID 0 maps via Linux user namespaces to the invoking host UID. Inside the container, `sudo` is effectively a no-op (process is already root); outside, the container holds no host privileges beyond what the invoking user already has.
- **Why Podman over Docker.** First-class rootless on Tumbleweed, no daemon, user-namespace is the default. Docker rootless is workable but fussier on SUSE.
- **No VM.** A VM would address C more thoroughly but is mismatched to the user's threat ranking (C is medium, not paramount) and incurs ergonomic cost not justified by the marginal protection.

### 4.2 Container model

- **One persistent container per project.** Named per Section 5.1.
- Lifetime: `podman create` once; `podman start` to wake; `podman exec` to enter. Container survives host reboots; the wrapper detects stopped state and restarts.
- `claude-sandbox stop` is reversible (preserves writable layer + volumes).
- `claude-sandbox down` destroys the container and named home volume. The project directory on host is never touched.

### 4.3 Base image

- Minimal Debian (bookworm-slim) + Claude Code + `git` + `sudo` + `curl` + `ca-certificates` + `openssh-client` + `build-essential` + `pkg-config` + `jq` + `direnv` + `tailscale` + the `cs` companion binary (same binary as `claude-sandbox`, two entry points distinguished by `argv[0]`; see Section 7.1).
- ~280 MB. Versioned (`claude-sandbox:0.1`). Rebuilds when `~/.config/claude-sandbox/Dockerfile` changes (Dockerfile hashed into the image tag).
- The Dockerfile lives at `~/.config/claude-sandbox/Dockerfile`, user-editable. The binary ships a "write defaults" command for first-run setup.

### 4.4 Mount policy

Defaults at every container start:

| Host path | Container path | Mode |
| --- | --- | --- |
| `$PWD` (project root) | `/work` | rw |
| `~/.claude/` | `/root/.claude` | rw |
| `~/.gitconfig` | `/root/.gitconfig` | ro |
| named volume `cs-<container-name>-home` | `/root` (overlay for everything else) | rw |

The `<container-name>` in the volume name matches the container's name from Section 5.1.

Not mounted, ever, unless explicitly declared per-project: `~/.ssh`, `~/.aws`, `~/.config/*` (other than the explicit list above), `~/Documents`, etc.

`~/.claude/` is intentionally rw so skills, agents, settings, credentials, and auto-memory are bidirectional. Tradeoff documented in Section 8 (concurrent writes).

### 4.5 Network

- Default `bridge`, full outbound.
- Inbound: none unless `ports` declared in config. Host-port collisions auto-shift (Section 5.3).
- Tailscale: each container can be its own tailnet device via own `tailscaled --tun=userspace-networking`, authenticated by `TS_AUTHKEY`. Host is not affected (host need not be on tailnet at all). Opt-in per project.

### 4.6 Credentials

- **SSH:** forward host SSH agent socket by default (`SSH_AUTH_SOCK` env + bind-mount the socket). Container can use loaded keys; never sees the key files. Disable per-project via `ssh_agent = false`.
- **Everything else (cloud CLIs, API tokens, Pulumi, etc.):** default deny. Per-project opt-in via `mount` and `env_passthrough` in `.claude-sandbox.toml`.

## 5. Lifecycle & CLI

### 5.1 Project identification and naming

- Container name derived from the project path's component sequence below `$HOME`, lowercased, `/` and whitespace → `-`. Example: `~/Documents/projects/spire` → `documents-projects-spire`. Paths outside `$HOME` use `root-<absolute-path>` with the same normalization.
- **The derived name is written to `.claude-sandbox.toml` on first container creation.** From that point, the name lives in the file. Moving the project directory preserves the association because the toml moves with it.
- If two paths derive identical names, the second container appended `-<8charhash>` automatically.
- Project root discovery: walk up from `$PWD` looking for `.claude-sandbox.toml` first, then `.git`; whichever is closer is the project root.

### 5.2 Commands

All commands resolve the current project from CWD automatically.

| Command | Behavior |
| --- | --- |
| `claude-sandbox` (alias `start`) | Ensure container exists → ensure running → run `claude` inside. The common path. |
| `claude-sandbox shell` | Same lifecycle, drop into `bash`. |
| `claude-sandbox stop` | `podman stop` (reversible, preserves state). |
| `claude-sandbox down` | Destroy container + named home volume. Project dir untouched. |
| `claude-sandbox status` | Show container existence, run state, mounts, age, package list. |
| `claude-sandbox ls` | List all `cs-*` containers across all projects. `--orphans`, `--size` modifiers. |
| `claude-sandbox rebuild` | Rebuild base image. Existing containers keep their image pin. `--recreate` offers to recreate each. |
| `claude-sandbox logs` | Tail container logs. |
| `claude-sandbox rename <new-name>` | Atomic rename: `podman rename` + update `name` field in `.claude-sandbox.toml` (via `toml_edit` to preserve formatting) + update registry cache. The toml update is performed host-side and is **not blocked by `agent_writable = false`**, since the wrapper is running as the user, not from inside the container. |
| `claude-sandbox migrate <new-path>` | Re-associate an orphan container with a new path. An "orphan" is a `cs-*` container whose recorded path no longer exists (e.g. project moved while toml was absent). |

### 5.3 Worktree picker

When `claude-sandbox` (no args) is invoked:

- **No worktrees exist:** skip the menu, launch Claude in the main checkout. Zero friction.
- **Worktrees exist:** show a numbered picker with status annotations.

Example picker output:

```
$ claude-sandbox
Project: spire  (container: documents-projects-spire, running)

  1. main              [available]
  2. fix-auth          [in-use: claude on host PID 12345 since 14:02]
  3. refactor-db       [available]
  4. polish-ui         [stale claim — will reclaim]
  5. + new worktree
  q. quit

> _
```

Flags to skip the picker: `--main`, `-w <name>`, `--no-menu` (errors if picker would have shown).

**In-use detection.** Each invocation writes `.worktrees/<name>/.cs-session` containing `{host_pid, started_at, container_exec_id}`. Removed on clean exit. If a claim file exists but the recorded host PID is no longer alive (e.g. terminal was killed), the claim is considered stale; the new invocation overwrites it with its own and proceeds. Active claims block reuse unless `--force` is passed.

### 5.4 In-container companion (`cs`)

A second entry point of the same binary, available inside the container as `/usr/local/bin/cs`. Used by Claude itself when it decides it wants a worktree.

| Command | Behavior |
| --- | --- |
| `cs worktree add <name> [--branch <branch>]` | `git worktree add /work/.worktrees/<name>` + run `worktree_setup` hooks from `.claude-sandbox.toml` with `CS_WORKTREE_NAME` exported. |
| `cs worktree ls` | List worktrees with status. |
| `cs worktree rm <name>` | `git worktree remove` + prune. |
| `cs worktree current` | Print the worktree name (or `main`) for the current CWD. |
| `cs status` | Container-side equivalent of `claude-sandbox status`. |

## 6. `.claude-sandbox.toml` schema

All fields optional. Without the file, defaults apply.

Auto-created on first `claude-sandbox start` if missing, populated only with `name` and a header comment. The file is the contract between user and agent; **mounted read-only into the container by default.** Opt out via `agent_writable = true`.

```toml
# claude-sandbox config — see `claude-sandbox docs`

# ---- Identity ----
name = "documents-projects-spire"

# ---- Permissions ----
agent_writable = false             # default: false; when true, mount this file rw

# ---- Image ----
image = "claude-sandbox:0.1"       # default: pin from global config

# ---- Additional bind-mounts (beyond defaults from Section 4.4) ----
mount = [
  { host = "~/.config/pulumi", container = "/root/.config/pulumi", ro = true },
  { host = "/data/blobs",      container = "/blobs",               ro = false },
]

# ---- Env ----
env_passthrough = ["TS_AUTHKEY", "PULUMI_ACCESS_TOKEN", "OPENAI_API_KEY"]
env             = { CARGO_TERM_COLOR = "always", RUST_BACKTRACE = "1" }
env_file        = ".claude-sandbox.env"   # gitignored by convention

# ---- Network ----
ssh_agent = true                   # forward host SSH agent socket
network   = "bridge"               # bridge | host | none
ports = [
  "5173:5173",                     # host port preferred; auto-shift on collision
  "!8080:8080",                    # strict; errors on collision
  ":3000",                         # any ephemeral host port → container 3000
]

[tailscale]
enabled     = false
hostname    = "cs-spire"           # default: container name
authkey_env = "TS_AUTHKEY"         # which host env var holds the auth key

# ---- GPU (opt-in for ML workflows, e.g. ComfyUI) ----
gpu = false                        # true → --device nvidia.com/gpu=all (CDI)

# ---- Lifecycle hooks ----
setup = [                          # runs ONCE on container creation
  "apt-get install -y build-essential libssl-dev pkg-config",
  "curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y",
]
on_start = []                      # runs each container start
on_stop  = []                      # runs at stop (best-effort)
worktree_setup = [                 # runs after `cs worktree add <name>`
  "echo 'CARGO_TARGET_DIR=/work/.worktrees/$CS_WORKTREE_NAME/target' > .envrc",
  "direnv allow . || true",
]

# ---- Resource limits ----
[limits]
memory = "16g"
cpus   = 8
```

### 6.1 Parse rules

- `~` and `$VAR` expanded in host paths.
- `env_passthrough` silently skips vars not set on host.
- `env_file` is project-relative; ignored if missing.
- Validation runs on load; failure points at field + line.
- Hooks execute via `bash -c`. Non-zero exit on `setup` / `worktree_setup` aborts the operation; non-zero on `on_start` / `on_stop` warns but continues.
- Exported in hook env: `CS_PROJECT_NAME`, `CS_PROJECT_PATH`, `CS_WORKTREE_NAME` (only in `worktree_setup`).
- Port syntax: `"H:C"` preferred-with-shift, `"!H:C"` strict, `":C"` ephemeral.

### 6.2 Global defaults

`~/.config/claude-sandbox/config.toml` accepts the same schema; values apply to every project that does not override them. Example use: globally `setup = ["apt-get install -y ripgrep fd-find"]`.

## 7. Implementation

### 7.1 Language

Rust. Single static binary. Crate stack:

- `clap` (derive) — subcommands and flags.
- `serde` + `toml` (parse) + `toml_edit` (round-trip preserving comments) — config.
- `dialoguer` — the worktree picker.
- `nix` — `execvp` for replacing the wrapper process with `podman exec` (clean signal forwarding).
- `which` — locate podman.
- No async runtime.

Same binary serves two entry points: `claude-sandbox` (host) and `cs` (in-container), distinguished by `argv[0]`. `cs` is a symlink in the base image.

### 7.2 Layout

Host:

```
~/.local/bin/claude-sandbox             # the binary (host entry point)
~/.config/claude-sandbox/
  config.toml                           # global defaults
  Dockerfile                            # base image recipe, user-editable
~/.local/share/claude-sandbox/
  registry.json                         # path → container-name cache (rebuildable)
~/.cache/claude-sandbox/                 # transient
```

Per-project:

```
<project>/.claude-sandbox.toml          # version-controlled
<project>/.claude-sandbox.env           # gitignored
<project>/.worktrees/<name>/            # git worktrees (gitignored)
<project>/.worktrees/<name>/.cs-session # claim file (PID, exec_id, started_at)
```

### 7.3 Container engine integration

Shell out to `podman` CLI with `--format json` for queries. No wrapping of the REST API socket. `-v` prints every podman invocation; `-vv` adds resolved TOML and mount lists.

### 7.4 Base image (sketch)

```dockerfile
FROM debian:bookworm-slim

RUN apt-get update && apt-get install -y --no-install-recommends \
      ca-certificates curl git sudo bash openssh-client \
      build-essential pkg-config jq direnv tailscale \
    && rm -rf /var/lib/apt/lists/*

# Claude Code per Anthropic's official installer
RUN curl -fsSL https://claude.ai/install.sh | bash

COPY claude-sandbox /usr/local/bin/claude-sandbox
RUN chmod +x /usr/local/bin/claude-sandbox \
 && ln -s claude-sandbox /usr/local/bin/cs

WORKDIR /work
ENTRYPOINT ["/bin/bash", "-l"]
```

### 7.5 Process model

`claude-sandbox start` performs: resolve project → ensure container exists → ensure running → `execvp("podman", ["exec", "-it", name, "claude"])`. The wrapper replaces itself; no supervisor process, no daemons.

### 7.6 Build / install

- Repo at `~/Documents/projects/claude-sandbox`.
- `make install` builds with `cargo build --release`, copies the binary to `~/.local/bin/claude-sandbox`, and copies default `Dockerfile` + `config.toml` to `~/.config/claude-sandbox/` (no-clobber). The `cs` symlink to the same binary is created only inside the container image (Section 7.4 / 4.3), not on the host.
- `claude-sandbox build` triggers image build the first time it's needed.

## 8. Edge cases

| Case | Handling |
| --- | --- |
| Project moved on host | Name lives in `.claude-sandbox.toml`; association is path-independent. Only an issue if user deletes the toml after a move — in which case `claude-sandbox migrate <new-path>` re-points. |
| `cd` into project subdirectory | Wrapper walks up looking for `.claude-sandbox.toml` first, then `.git`; closer wins. |
| Two paths derive identical name | First-created wins; second gets `-<8charhash>` suffix. |
| Image update | Existing containers keep their image pin. `rebuild` rebuilds image only; `rebuild --recreate` offers to recreate each. |
| Host reboot mid-session | Wrapper detects stopped state and restarts. No orphaned state. |
| `~/.claude/MEMORY.md` concurrent writes (multiple sandboxes in parallel) | Last-write-wins. Acceptable per user. Worth revisiting if it bites in practice. |
| Disk usage | `ls --size` shows per-container layer + volume. `down` reclaims. Image is shared, doesn't multiply. |
| Mount target overlap | Validated at start; clear error naming the colliding mounts. |
| Empty / broken `.claude-sandbox.toml` | Validation fails fast with field + line number; refuses to start. |
| `git config --global` inside container | `~/.gitconfig` is ro; writes fail. Acceptable — git config is host-side. Use local git config or env in toml if a project needs different identity. |
| Tailscale device churn after `down` | Old machine appears as offline in tailnet admin. User deletes manually. Acceptable. |
| Port collision (default `"H:C"`) | Auto-shift to next free host port (up to +20, then OS ephemeral). Mapping reported at start and visible in `status`. |
| Port strict (`"!H:C"`) | Podman errors as today; user fixes in toml. |
| Cargo target dir per worktree | Default `worktree_setup` writes `.envrc` with per-worktree `CARGO_TARGET_DIR` + `direnv allow`. Each worktree builds independently. |

## 9. Build order

1. Bare wrapper: `start` / `shell` / `stop` / `down` / `status` with hardcoded image, no config file. Solidify the path → name pipeline and podman shell-out.
2. Add `.claude-sandbox.toml` parsing + auto-creation + global config + mounts + env + hooks.
3. Add the `cs` in-container companion + worktree commands.
4. Add the picker UI when worktrees exist + claim-file logic.
5. Add Tailscale, GPU, ports (with auto-shift), `rename`, `migrate`.
6. Polish: `ls --orphans --size`, `rebuild --recreate`, error messages, README/docs.

## 10. Open questions (deferred)

1. Bind-mount host dotfiles (`~/.bashrc`, etc.) into container by default, or opt-in only? Current plan: opt-in only via `mount`.
2. Multi-host distribution of `~/.config/claude-sandbox/` (`config.toml` + `Dockerfile`). Out of scope; layout makes it dotfiles-friendly.
3. First-run image build: prompt before building, or build silently with progress? Current plan: silent-with-progress.
4. Telemetry / debug capture beyond `-v` / `-vv`. None planned.
